# Standard Poster Design System Plan

## 0. Highest Principle: OTR Story Design Language

The goal of Standard Poster is not only to generate a beautiful poster.

The deeper goal is:

> Let any Journey moment become part of a stable, recognizable, and designed visual language.

When a user creates 100 Stories, the result should not feel like 100 unrelated templates. It should feel like OTR has its own Story Design Language.

Apple has a design language. National Geographic has a design language. OTR should have a Journey-native story language: real, restrained, emotional, mobile-first, and trustworthy.

This means Standard Poster should optimize for:

- Visual consistency across many generated Stories.
- Real Journey material over decorative invention.
- Strong composition over template variety.
- Confident whitespace over filling every area.
- Few words when the photo already carries the emotion.
- Repeatable grammar that can create many expressions without copying entire templates.

## 1. Standard Poster Definition

Standard Poster is the default creation level for OTR Stories.

It is not Pro, Studio, Discover content, or AI-generated artwork. It is the reliable default artifact created from a Journey Memory Shot.

Standard Poster uses:

- Real user Journey photos and existing Journey data.
- Story Script generated by the existing Memory Shot flow.
- Design Decision produced by deterministic rules.
- A local renderer that outputs image files.
- Existing Render Storage Router for storage.

Standard Poster does not call an AI image generation model and does not generate fake travel scenes.

The goal is stable, authentic, beautiful, and shareable output.

Standard Poster is not a template system. It is a layout engine based on poster grammar.

Templates are fixed pages. Grammar is a set of composable rules. OTR should build the grammar.

## 2. Quality Goals

Standard Poster must be good enough that a user would consider sharing it to WeChat Moments, Xiaohongshu, Instagram, or a private group chat.

It must:

- Feel like a travel story poster, not a debug card.
- Avoid a PowerPoint-like layout.
- Avoid large empty regions unless intentional in a magazine layout.
- Avoid sensitive spending details, receipts, or raw ledger amounts.
- Avoid text overlap in every supported layout.
- Never show `MEMORY SHOT` as user-visible copy.
- Prioritize real photos when usable photos exist.
- Keep text restrained and story-like.
- Preserve date, place, and journey context without exposing database-like fields.
- Use preview/thumbnail assets in browsing surfaces, not Google Drive originals.
- Allow intentional whitespace and silence.
- Allow one strong sentence instead of forcing three story blocks.

## 3. Poster Composition Grammar

Poster is not Template.

Poster is Grammar.

Every Standard Poster is composed from design components:

- Hero Component
- Title Component
- Story Component
- Metadata Component
- Footer Component
- Overlay Component

Layout is only one possible composition of these components. Future layouts should not copy a full HTML/SVG template. They should combine the same components with different positions, density, scale, rhythm, and rules.

Example composition:

```txt
Hero.photo.full_bleed
+ Title.huge.bottom_left
+ Story.minimal.one_line
+ Metadata.date_place.compact
+ Overlay.bottom_gradient
+ Footer.logo.bottom_left
```

Another composition:

```txt
Hero.collage.board
+ Title.left.editorial
+ Story.cards.two_beats
+ Metadata.people_date
+ Overlay.none
+ Footer.logo_rule
```

The renderer should eventually think in component slots, not page templates.

### Grammar Components

#### Hero Component

Hero defines the primary visual surface.

Allowed forms:

- `photo`: one real Journey photo.
- `video_frame`: a frame from a user video, if available later.
- `map`: route or location-driven visual.
- `collage`: multiple real Journey photos.
- `none`: text-first fallback.

#### Title Component

Title defines hierarchy and emotional entry.

Allowed forms:

- `huge`
- `left`
- `center`
- `bottom`
- `overlay`
- `quiet_editorial`

#### Story Component

Story defines content density and reading mode.

Allowed forms:

- `quote`
- `one_line`
- `paragraph`
- `timeline`
- `cards`
- `caption_stack`

#### Metadata Component

Metadata gives context without feeling like database output.

Allowed forms:

- `date_place`
- `journey`
- `people`
- `route`
- `hidden`

#### Footer Component

Footer defines ending and brand presence.

Allowed forms:

- `logo`
- `journey_name`
- `logo_rule`
- `qr_future`
- `nothing`

#### Overlay Component

Overlay gives readability and atmosphere.

Allowed forms:

- `bottom_gradient`
- `top_gradient`
- `full_vignette`
- `solid_panel`
- `blur_panel`
- `none`

### Layout Presets, Not Templates

The five Standard Layouts below are not full templates. They are first-party grammar presets.

Each preset should be implemented as a composition of shared components. Adding a future visual style should usually mean adding a new preset or component variant, not copying a whole renderer file.

## 4. Narrative Density

Narrative Density controls how much text a poster uses.

This is separate from layout. The same layout can render minimal, normal, or rich content.

### minimal

Use when the photo is strong and the moment is clear.

Structure:

- Visual-first hero.
- One title or one sentence.
- Optional compact date/place.
- Logo.

Example:

```txt
Golden Circle.
```

Minimal posters should be allowed. They are often the most premium outputs.

### normal

Use for most Daily Best Moments.

Structure:

- Hero.
- Title.
- Metadata.
- 1-2 story beats.
- Footer branding.

### rich

Use when the Journey data has a meaningful sequence.

Structure:

- Hero.
- Title.
- Metadata.
- 3-4 story beats or timeline.
- Optional route/location block.
- Optional ending.

Rich should not mean crowded. It means the story has enough structure to deserve more content.

### Density Decision

The Story Script and Design Decision may suggest density. The renderer must support it.

AI may decide that today only needs one sentence. The renderer must not force every poster into three blocks.

Deterministic fallback:

- Strong photo + simple day: `minimal`.
- Ordinary photo + enough story data: `normal`.
- Route/multi-stop/multi-person day: `rich`.
- Weak photo or no photo: `normal` or `rich` depending on text quality.

## 5. Hero Roles

Photos and visual assets are not interchangeable. A poster needs roles.

### Hero

The main cover visual.

Hero should:

- Carry the emotion of the poster.
- Be large enough for the selected layout.
- Be renderable by the server.
- Work with title and overlay.

Examples:

- Iceland waterfall.
- Golden hour road.
- Group photo with clear faces.

### Support

Secondary visual evidence.

Support assets should:

- Add detail to the story.
- Support collage, cards, or story sections.
- Not compete with Hero.

Examples:

- Group selfie.
- Meal photo.
- Hotel exterior.
- Hiking stop.

### Texture

Atmosphere or background material.

Texture assets should:

- Add mood without needing to be read.
- Work behind overlays or panels.
- Be safe to crop aggressively.

Examples:

- Sky.
- Water surface.
- Road texture.
- Blurred city lights.

The same Journey day might use:

```txt
Hero: waterfall
Support: group photo
Texture: sky
```

The Design Decision should tell the renderer which asset plays which role.

## 6. First Five Standard Layout Presets

The first version should support only five layouts. The priority is quality and repeatability, not template volume.

### cinematic_full_bleed

Full-screen photo grammar preset.

Structure:

- Hero Component: `photo.full_bleed`.
- Overlay Component: `bottom_gradient` or `full_vignette`.
- Title Component: `huge.bottom_left`.
- Metadata Component: `date_place.compact`.
- Story Component: `one_line` or `paragraph`.
- Footer Component: `logo.bottom_left`.

Best for:

- High-quality landscape photos.
- Strong atmosphere.
- Sunrise, sunset, road, mountain, lake, beach, skyline, street scene.

### hero_top_story_bottom

Classic image-first story grammar preset.

Structure:

- Hero Component: `photo.top`.
- Overlay Component: `top_gradient` or `none`.
- Title Component: `left` or `overlay`.
- Metadata Component: `date_place`.
- Story Component: `cards` or `paragraph`.
- Footer Component: `logo_rule`.

Best for:

- One usable photo that is not strong enough for full bleed.
- Busy images that need a clean text area.
- Mixed daily summary stories.

### magazine_white_space

Quiet editorial layout.

Structure:

- Hero Component: `photo.small`, `texture`, or `none`.
- Overlay Component: `none`.
- Title Component: `quiet_editorial`.
- Metadata Component: `date_place`.
- Story Component: `quote`, `paragraph`, or `caption_stack`.
- Footer Component: `logo`.

Best for:

- Weak or unavailable photos.
- Sparse Journey data.
- More reflective story copy.

### collage_memory_board

Social memory layout.

Structure:

- Hero Component: `collage`.
- Overlay Component: `none` or `soft_panel`.
- Title Component: `left` or `bottom`.
- Metadata Component: `people_date`.
- Story Component: `caption_stack` or `cards`.
- Footer Component: `logo_rule`.

Best for:

- Three or more usable photos.
- People/group-heavy days.
- Food, friends, small moments, shared activities.

### route_story_card

Movement and location-driven layout.

Structure:

- Hero Component: `map` or `photo + route`.
- Overlay Component: `panel`.
- Title Component: `left`.
- Metadata Component: `route`.
- Story Component: `timeline`.
- Footer Component: `logo`.

Best for:

- Route data.
- Multiple captured locations.
- Planner items with a clear sequence.
- Driving, hiking, city walking, road trip days.

## 7. Renderer Components

The renderer should not rewrite every layout from scratch. Standard layouts should compose shared renderer components.

Core components:

- `HeroImage`: Loads and crops a selected real image.
- `HeroSurface`: Chooses photo, map, collage, texture, or text-first surface.
- `GradientOverlay`: Adds readable image overlays.
- `TitleBlock`: Renders title with controlled line count and font scale.
- `MetadataLine`: Renders date, place, people, or journey context.
- `StoryBeats`: Renders 1-4 compact story lines.
- `StoryBlock`: Renders quote, one-line, paragraph, cards, timeline, or caption stack.
- `PhotoGrid`: Renders multi-photo collage layouts.
- `RouteBlock`: Renders route/location-driven visual structure.
- `BrandingFooter`: Renders OTR branding and footer rule.
- `SafeArea`: Defines protected bounds for text, image, and logo.
- `TextFit`: Applies text measurement, wrapping, truncation, and font-size fallback.
- `WhitespaceBudget`: Reserves intentional empty space and prevents content stuffing.

Shared components should keep layout behavior predictable across templates and reduce regressions when typography changes.

## 8. TextFit And Whitespace Rules

TextFit is mandatory for every Standard Poster layout.

Baseline limits:

- Title: maximum 2 lines for full-bleed and route layouts, maximum 3 lines for magazine layout.
- Subtitle: maximum 1-2 lines.
- Metadata line: maximum 1 line.
- Story beat: maximum 2 lines per beat.
- Total story beats: normally 1-3, maximum 4 only in text-first fallback.
- Ending/quote: maximum 1-2 lines, optional.

Density-specific limits:

- `minimal`: title or one story line only; no story cards.
- `normal`: title, metadata, and 1-2 story beats.
- `rich`: title, metadata, and up to 4 story beats or timeline steps.

Overflow behavior:

1. Wrap text within the layout safe area.
2. Reduce font size within the layout’s allowed range.
3. Reduce number of story beats.
4. Truncate final visible line with ellipsis.
5. Drop optional ending before dropping core title/subtitle.

Hard rules:

- Text must never overlap branding/footer.
- Text must never overflow the poster crop.
- Text must never overlap another text block.
- Text must remain readable over images through gradient or solid panels.
- Logo/footer safe area is reserved before story text is placed.
- Layout must preserve intentional whitespace when the density is `minimal`.
- Renderer must never fill empty space merely because more text exists.

## 9. Photo Selection And Asset Role Rules

Phase 1 photo selection must be deterministic and should not call an LLM.

Use existing photo index and media metadata where available.

Every selected visual asset should be assigned a role:

- `hero`: primary cover visual.
- `support`: secondary story visual.
- `texture`: atmosphere/background visual.

Selection rules:

- Prefer `previewUrl` over thumbnail URLs for render quality.
- Prefer images that can be fetched by the renderer.
- Prefer vertical images for `cinematic_full_bleed`.
- Prefer horizontal images for `hero_top_story_bottom`.
- Prefer landscape/scene tags for `cinematic_full_bleed`.
- Prefer people/group metadata for `collage_memory_board`.
- Prefer multiple distinct photos for collage.
- Prefer atmospheric but non-essential images as `texture`.
- Do not use the same asset for every role unless there is only one usable photo.
- Avoid very small, broken, duplicated, or unavailable images.
- If there are no usable photos, fallback to `magazine_white_space`.

Useful metadata signals:

- Width and height.
- Orientation.
- Scene tags.
- Face/person metadata.
- Sharpness/blur score, if available.
- Capture time.
- Location proximity to the story date.
- Existing AI photo summary, if already available.

## 10. Layout Selection Rules

Layout selection is deterministic in the first implementation.

Priority:

1. High-quality landscape photo exists: `cinematic_full_bleed`.
2. Three or more usable photos exist: `collage_memory_board`.
3. Route or strong location sequence exists: `route_story_card`.
4. One ordinary usable photo exists: `hero_top_story_bottom`.
5. No usable photo exists: `magazine_white_space`.

Tie-breakers:

- If a route exists but the photo is visually outstanding, prefer `cinematic_full_bleed`.
- If people/group content is the story focus, prefer `collage_memory_board`.
- If all photos are weak, prefer `magazine_white_space` even if photos exist.
- If selected image fails at render time, fallback to `magazine_white_space`.

The output of layout selection is a grammar preset plus component decisions, not a standalone template.

## 11. Design Decision Schema

Design Decision is the explicit contract between content generation and rendering.

```ts
type StandardPosterDesignDecision = {
  layoutKey:
    | "cinematic_full_bleed"
    | "hero_top_story_bottom"
    | "magazine_white_space"
    | "collage_memory_board"
    | "route_story_card";
  variantKey: string;
  composition: {
    hero: "photo" | "video_frame" | "map" | "collage" | "none";
    title: "huge" | "left" | "center" | "bottom" | "overlay" | "quiet_editorial";
    story: "quote" | "one_line" | "paragraph" | "timeline" | "cards" | "caption_stack";
    metadata: "date_place" | "journey" | "people" | "route" | "hidden";
    footer: "logo" | "journey_name" | "logo_rule" | "qr_future" | "nothing";
    overlay: "bottom_gradient" | "top_gradient" | "full_vignette" | "solid_panel" | "blur_panel" | "none";
  };
  narrativeDensity: "minimal" | "normal" | "rich";
  themeKey: "warm_editorial" | "cool_cinematic" | "neutral_clean" | "high_contrast";
  heroAssetId: string | null;
  supportAssetIds: string[];
  textureAssetIds: string[];
  assetRoles: Array<{
    assetId: string;
    role: "hero" | "support" | "texture";
    reason: string;
  }>;
  cropMode: "cover_center" | "cover_top" | "cover_bottom" | "contain" | "manual";
  overlayMode: "none" | "top" | "bottom" | "full_vignette" | "panel";
  title: string;
  subtitle: string;
  storyBeats: string[];
  metadata: {
    dateText?: string;
    placeText?: string;
    authorText?: string;
    peopleText?: string;
    routeText?: string;
    sourceTemplateKey?: string;
  };
  branding: {
    label: "OTR" | "OTR STORY" | "Journey Story";
    placement: "top_left" | "bottom_left" | "bottom_center";
  };
  safetyFlags: {
    hasSensitiveExpense: boolean;
    hasPrivateLocation: boolean;
    hasWeakPhotoQuality: boolean;
    hasMissingHeroAsset: boolean;
    textOnlyFallback: boolean;
    containsDebugCopy: boolean;
    shouldUseMinimalCopy: boolean;
  };
};
```

The renderer should store the final Design Decision in artifact metadata or manifest so outputs can be inspected and regenerated.

## 12. Renderer Output

The renderer continues to generate:

- `original_render.png`
- `preview_render.webp`
- `thumbnail_render.webp`

It must continue to use the existing Render Storage Router.

Storage rules remain unchanged:

- Original: Google Drive first, fallback private Supabase original storage.
- Preview: media server first, fallback Supabase.
- Thumbnail: media server first, fallback Supabase.

Do not modify the storage router in this phase.

## 13. Acceptance And Debug Preview

A local preview/debug tool is required before switching Daily Best Moments to the new renderer.

The tool should support:

- Input `memory_shot_id`.
- Optional forced `layoutKey`.
- Render all five Standard layouts for the same Memory Shot.
- Output preview files locally or to a debug storage location.
- Print chosen Design Decision for each layout.
- Print selected hero/support asset ids.
- Print texture asset ids and asset roles.
- Print narrative density.
- Print grammar composition.
- Print render warnings.

Suggested command shape:

```bash
node scripts/preview-standard-posters.mjs \
  --memory-shot-id <memory_shot_id> \
  --all-layouts
```

Forced layout example:

```bash
node scripts/preview-standard-posters.mjs \
  --memory-shot-id <memory_shot_id> \
  --layout cinematic_full_bleed
```

This tool is important because visual quality cannot be validated from typecheck alone. We need fast side-by-side comparison before making any layout the default.

## 14. Implementation Steps

### Phase 9B: Standard Poster Renderer Component Skeleton

- Add shared renderer component helpers.
- Add Design Decision type.
- Add Poster Composition Grammar types.
- Add Narrative Density support.
- Add Hero/Support/Texture asset roles.
- Add deterministic design decision builder.
- Keep existing renderer output path unchanged.
- Do not switch Daily Best Moments yet.

### Phase 9C: Implement cinematic_full_bleed

- Full-bleed image layout.
- Strong gradient overlay.
- Large title and short story text.
- Strict TextFit.

### Phase 9D: Implement hero_top_story_bottom

- Top image and bottom story card.
- Clean fallback when image fails.
- Good default for ordinary photos.

### Phase 9E: Implement collage_memory_board

- 1 main image plus 2-4 support images.
- Good spacing and photo rhythm.
- People/group-friendly copy placement.

### Phase 9F: Implement magazine_white_space

- Text-first editorial fallback.
- Optional small image.
- Carefully controlled whitespace.

### Phase 9G: Implement route_story_card

- Route/location-driven visual block.
- Simple stop/story layout.
- No map vendor dependency in the first pass unless already available.

### Phase 9H: Layout Preview/Debug Script

- Render five layouts for one `memory_shot_id`.
- Save or upload previews.
- Print Design Decision and storage/debug paths.

### Phase 9I: Switch Daily Best Moments Default To New Renderer

- Keep legacy fields and artifact dual write.
- Make Daily Best Moments use Standard Poster Design Decision.
- Let Daily Best Moments use `minimal`, `normal`, or `rich` density depending on the Story Script and selected assets.
- Keep fallback to old render path if needed.
- Verify against local smoke run and manual visual checklist.

## 15. Current Non-Goals

This phase does not include:

- AI image generation.
- Enhanced or Studio poster tiers.
- Discover.
- New Story templates.
- Recommendation engine changes.
- Motion Story changes.
- Major Stories page changes.
- Render Storage Router changes.
- Google Drive storage behavior changes.
- Public sharing changes.
