# Story Page V1

This document records the first product structure for the Journey Story page.

Story Page V1 is a product cleanup layer on top of the existing Memory Shot MVP. It does not rename database tables, workers, renderer code, artifacts, or APIs.

## Naming

User-facing name:

- Chinese: 故事
- English: Stories

User-facing content should use Story / Stories. The technical object remains Memory Shot / `memory_shots`.

This keeps the product language simple while preserving the existing implementation model:

```text
Story = Memory Shot + Artifacts
```

## Product Positioning

The Story page is the Journey-level place for browsing and creating narrative outputs from trip activity.

It is not a leaderboard page, a debug page, or an admin surface. It should feel like a lightweight story gallery and creation entry point.

## Page Structure

Story Page V1 has three visible sections.

### 1. My Stories

Shows stories generated by the current user.

Current behavior:

- Displays the user's generated Memory Shots as Story cards.
- Uses a horizontal rail.
- Uses a vertical poster-shaped cover card.

Future behavior:

- Include stories the user saved or favorited.
- Support filtering by created, saved, shared, or downloaded.

### 2. Latest Stories

Shows the latest stories generated by other Journey members.

Current behavior:

- Avoids duplicating My Stories when the current user is the only author.
- Shows an empty state such as "No new stories from other members yet" when there are no other member stories.
- Uses the same cover card as My Stories.

Future behavior:

- Include member attribution more prominently.
- Support story freshness, unread state, or lightweight reactions.

### 3. Worth Creating

Shows system-recommended story ideas that the user can generate.

Current behavior:

- Displays a small set of creation ideas.
- Daily Best Moments is the only active creation path.
- Clicking Create starts the existing background generation flow.

Future behavior:

- Convert Journey signals into richer story ideas.
- Use ranking, ledger, route, photo, people, and memory signals as inputs.
- Support multiple story templates once the MVP path is stable.

## Story Card

All Story cards use a vertical poster ratio.

A card should show:

- Cover, thumbnail, or preview image
- Title on the cover
- Date or subtitle
- Author
- Status
- Visibility
- Poster / Motion Story indicators when available

The cover image is treated as visual artwork. Users do not need to read all text inside the generated poster from the rail.

Clicking a card opens a full-screen detail view.

## Story Detail

The detail view should feel like a work presentation, not a debug inspector.

V1 structure:

- Large poster preview at the top
- Title, subtitle, status, author, and visibility below
- Primary action: Open Story, when a Motion Story artifact is ready
- Secondary actions: Share, Favorite, Visibility, Download placeholder
- Debug details collapsed by default

Debug details may include render warnings, render errors, or generation failure summaries, but they should not be visible by default.

## Artifacts

Poster and Motion Story are both artifacts of the same Story.

Current artifact model:

```text
Memory Shot
-> Story Script / content
-> Poster artifact
-> Motion Story artifact
```

Poster:

- Static visual artifact.
- Used as cover, preview, thumbnail, and future download/share asset.

Motion Story:

- Browser-playable story artifact.
- Current version is a lightweight scroll story skeleton.

The Story page should use one unified Story card regardless of whether the Story has only a Poster, only a Motion Story, or both.

## Leaderboard Handling

The old Highlights leaderboard UI is hidden in Story Page V1.

The leaderboard data is not deleted. It should be treated as future recommendation material for Worth Creating.

Potential future story ideas from ranking signals:

- Best photographer of the trip
- Hiking hero
- Food explorer
- Most reliable driver
- Best group photo
- Most generous host

V1 does not implement these templates. It only hides the traditional leaderboard UI from the Story page.

## Stability Boundaries

Story Page V1 must not change:

- Memory Shot worker
- Renderer worker
- Artifact system
- Motion Story API
- Database tables or migrations
- Existing render/storage behavior

The page remains a presentation and creation entry layer over the existing Memory Shot system.

## Current Limitations

- Only Daily Best Moments can be generated.
- Motion Story is a skeleton, not video export.
- No Discover integration.
- No public share page.
- No long image download flow.
- No unread badge.
- No Admin UI for story templates or artifacts.
- Favorite, Share, Visibility, and Download actions are mostly placeholders.

## Follow-Up Optimization

Recommended next product refinements:

- Make Story cards smarter about author, date, and artifact availability.
- Add real Share and Favorite actions.
- Add Visibility management.
- Add dedicated full-screen Story detail route if modal limits become awkward.
- Convert leaderboard/ranking signals into Worth Creating recommendations.
- Add unread state for new member stories.
- Add grid poster and richer Motion Story layouts.
- Prepare Story cards for Discover once public authorization rules are ready.
